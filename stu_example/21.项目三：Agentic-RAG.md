# 第一课时：Agentic RAG 概述与项目架构

## 1.1 什么是 RAG？

### 传统 RAG 的工作流程

```
用户提问 → 向量检索（找到相关文档）→ 拼接 Prompt → LLM 生成回答
```

**核心思想**：给大模型"外挂知识库"，让它在生成回答前先检索相关知识。

### 传统 RAG 的局限性

```
┌─────────────────────────────────────────────────────────┐
│                    传统 RAG 的问题                        │
├─────────────────────────────────────────────────────────┤
│  ❌ 单一数据源：只能从向量数据库中检索                       │
│  ❌ 被动检索：每次都是"问题 → 检索 → 回答"的固定流程         │
│  ❌ 无法决策：不能根据问题类型选择不同的数据源                │
│  ❌ 无法组合：不能先查 A 再根据结果查 B                      │
│  ❌ 无状态管理：不能记住上下文，不能执行多步操作              │
└─────────────────────────────────────────────────────────┘
```

---

## 1.2 Agentic RAG —— 让 AI 学会"思考"

### 核心理念

```
传统 RAG：   用户 → [检索] → 大模型 → 回答
Agentic RAG：用户 → 大模型（思考）→ 决策 → 调用工具 → 获取结果 → 再思考 → ... → 回答
```

### Agentic RAG 的关键能力

```
┌──────────────────────────────────────────────────────────────┐
│                    Agentic RAG 的四大能力                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  🧠 自主决策          🔧 工具调用          🔄 多步推理        │
│  Agent 根据问题       Agent 可以调用        Agent 可以执行     │
│  类型，自主判断       多个不同的工具，      多步操作，每步      │
│  应该使用哪个         包括数据库查询、      的结果影响下一      │
│  工具来获取信息       邮件发送、API调用等   步的决策           │
│                                                              │
│  📊 多源融合                                                │
│  Agent 可以同时查询                                             │
│  图数据库、关系型数据库、                                         │
│  Redis 缓存等多种数据源                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 本项目中的 Agentic RAG 示例

**场景一：用户问"糖尿病有什么症状？"**

```
用户提问
    ↓
Agent 思考：这是一个医疗知识查询 → 需要使用 Neo4j 图数据库
    ↓
Agent 调用 find_data 工具，执行 Cypher 查询
    ↓
获取查询结果：多饮、多尿、体重下降、视力模糊...
    ↓
Agent 将结果整理成自然语言回答
    ↓
返回给用户
```

**场景二：用户请求"帮我发送验证码到邮箱"**

```
用户请求
    ↓
Agent 思考：需要发送验证码 → 先查用户邮箱 → 再发送验证码
    ↓
步骤1：调用 find_email 查询用户邮箱地址
    ↓
步骤2：调用 send_email 发送验证码到查到的邮箱
    ↓
步骤3：调用 verify_code 验证用户输入的验证码
    ↓
完成验证流程
```

### 为什么用 MCP 而不是直接调用？

```
❌ 直接调用方式：                    ✅ MCP 方式：

Agent 直接写 SQL                    Agent 调用 MCP 工具
    ↓                                    ↓
耦合度高，换数据库要改 Agent        MCP Server 封装具体实现
    ↓                                    ↓
安全风险，Agent 可能执行危险操作     MCP Server 可以做安全检查
    ↓                                    ↓
难以扩展，每个工具都要单独适配      统一协议，新增工具只需注册
```

## 2.1 MCP 协议简介

### 什么是 MCP？

**MCP（Model Context Protocol）** 是 Anthropic 提出的一种开放协议，用于标准化 AI 模型与外部工具、数据源之间的通信方式。

### 类比理解

```
MCP 之于 AI 工具调用  ≈  USB 之于计算机外设

• USB 定义了电脑与鼠标、键盘、打印机的连接标准
• MCP 定义了 AI 与数据库、API、文件系统的连接标准

没有 MCP = 每个设备都要专门的接口（混乱）
有了 MCP = 一个标准接口连接所有设备（统一）
```

### MCP 的三大核心概念

```
┌─────────────────────────────────────────────────────────────┐
│                      MCP 架构                                │
│                                                             │
│   ┌─────────────┐          ┌─────────────┐                 │
│   │   Client     │  ◄────► │   Server    │                 │
│   │  (AI 端)     │  HTTP    │  (工具端)    │                 │
│   │             │          │             │                 │
│   │ • 发起调用   │          │ • 暴露工具   │                 │
│   │ • 接收结果   │          │ • 执行操作   │                 │
│   └─────────────┘          │ • 返回结果   │                 │
│                            └─────────────┘                 │
│                                                             │
│   通信内容（JSON-RPC）：                                      │
│   - tools/list    →  列出所有可用工具                         │
│   - tools/call    →  调用指定工具                            │
│   - resources/read → 读取资源                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2.2 MCP Server：工具提供者

### 代码分析：`mymcp/MCPServer.py`

MCP Server 是本项目的核心，它将所有数据操作封装为标准的 MCP 工具。

```python
import pymysql
import redis
from fastmcp import FastMCP
import smtplib
from email.mime.text import MIMEText
from email.header import Header
import random
from langchain_neo4j import Neo4jGraph


mcp = FastMCP('cc')

# ===== Neo4j 配置 =====
NEO4J_URI = "bolt://localhost:7687"
NEO4J_USER = "neo4j"
NEO4J_PASSWORD = "rootroot"
neo_client = Neo4jGraph(NEO4J_URI, NEO4J_USER, NEO4J_PASSWORD)


@mcp.tool(
    name='find_email',
    description='通过用户名获取邮箱号',
)
async def find_email(sql: str, params: str = None) -> dict:
    """
    sql: SQL查询语句（必须是SELECT）
    params: 查询参数列表，例如 ["cc"]
    """
    # 安全限制：仅允许SELECT语句
    if not sql.strip().lower().startswith("select"):
        return {"result": "只允许执行 SELECT 查询语句，不可进行增删改操作！"}
    try:
        conn = pymysql.connect(
            host="127.0.0.1",
            port=3306,
            user="root",
            password="root",
            db="my_demo",
            charset="utf8mb4",
        )
        cursor = conn.cursor()
        cursor.execute(sql, params or ())  # 参数化查询
        columns = [desc[0] for desc in cursor.description]
        rows = cursor.fetchall()
        cursor.close()
        conn.close()
        result = [dict(zip(columns, row)) for row in rows]
        return {"result": result}
    except Exception as e:
        return {"result": str(e)}


@mcp.tool(
    name='send_email',
    description='发送邮件',
)
async def send_email(receiver: str) -> dict:
    """
    receiver: 收件人邮箱
    """
    # ====== 随机生成4位数验证码 ======
    code = ''
    for i in range(4):
        code += str(random.randint(0, 9))
    # ====== 邮件内容 ======
    sender = "1575422375@qq.com"  # 发件人邮箱
    password = ""  # 邮箱授权码或密码

    subject = "你好，你的验证来了"
    content = f"你的验证码为：{code}，有效时间60s"

    message = MIMEText(content, "plain", "utf-8")
    message["From"] = sender
    message["To"] = receiver
    message["Subject"] = Header(subject, "utf-8")

    # ====== SMTP 服务器配置 ======
    smtp_server = "smtp.qq.com"
    smtp_port = 587  # 通常 587 (TLS) 或 465 (SSL)

    try:
        # ====== 发送邮件 ======
        server = smtplib.SMTP(smtp_server, smtp_port)
        server.starttls()  # 启用 TLS 加密
        server.login(sender, password)
        server.sendmail(sender, receiver, message.as_string())
        server.quit()
        # ====== 验证码存入redis ======
        r = redis.Redis(
            host="127.0.0.1",  # Redis 服务器地址
            port=6379,  # Redis 端口号
            db=0,  # Redis 数据库编号
            password="root",  # Redis 密码
            decode_responses=True,  # 让 Redis 返回字符串，而不是字节
        )
        r.set(receiver, code, ex=60)
        return {"message": "邮件发送成功！"}
    except Exception as e:
        return {"result": str(e)}


@mcp.tool(
    name='verify_code',
    description='验证码验证',
)
async def verify_code(receiver: str, code: str) -> dict:
    """
    receiver: 收件人邮箱
    code: 验证码
    """
    r = redis.Redis(
        host="127.0.0.1",  # Redis 服务器地址
        port=6379,  # Redis 端口号
        db=0,  # Redis 数据编号
        password="root",  # Redis 密码
        decode_responses=True,  # 让 Redis 返回字符串，而不是字节
    )
    if r.get(receiver) == code:
        return {"result": "验证码验证成功！"}
    else:
        return {"result": "验证码验证失败！"}


@mcp.tool(
    name='find_data',
    description='在图数据库neo4j中查询数据',
)
def find_data(cypher: str, params: dict = None) -> dict:
    """
    执行 Neo4j Cypher 查询语句

    参数:
        cypher: Cypher 查询语句
        params: 可选参数字典

    返回:
        result: 查询结果列表
    """
    try:
        data = neo_client.query(cypher, params)
        return {"result": data}
    except Exception as e:
        return {"error": str(e)}


if __name__ == '__main__':
    mcp.run(
        transport="http",
        host="localhost",
        port=9000,
        show_banner=True,
        path='/mcp'
    )
```

### 工具注册一览表

| 工具名 | 功能 | 数据源 | 安全措施 | 返回类型 |
|--------|------|--------|----------|----------|
| `find_email` | 查询用户邮箱 | MySQL | 仅允许 SELECT，参数化查询 | `dict` |
| `send_email` | 发送验证码 | SMTP + Redis | 随机 4 位码，60s TTL | `dict` |
| `verify_code` | 验证验证码 | Redis | 比对后返回结果 | `dict` |
| `find_data` | 知识图谱查询 | Neo4j | Cypher 参数化查询 | `dict` |

### 关键设计要点

```
🔐 安全设计：
   • find_email 强制检查 SQL 必须以 SELECT 开头（不区分大小写）
   • 所有数据库操作使用参数化查询，防止 SQL/NoSQL 注入
   • 验证码使用 Redis TTL 机制，60 秒自动过期
   • 验证码通过 QQ 邮箱 SMTP TLS 加密发送

🧩 解耦设计：
   • Agent 不需要知道 MySQL 的连接信息
   • Agent 不需要知道 SMTP 的配置和授权码
   • 更换数据源只需修改 MCP Server，不影响 Agent
   • 每个工具通过 @mcp.tool() 装饰器注册，统一管理
```

## 2.3 MCP Client：工具调用者

### 代码分析：`mymcp/MCPClient.py`

MCP Client 是 LangChain 工具和 MCP Server 之间的桥梁，通过 HTTP 协议调用 MCP Server 上的工具。

```python
import asyncio
from fastmcp import Client
import warnings

# 忽略警告
warnings.filterwarnings("ignore", category=DeprecationWarning)
MCP_SERVER_URL = 'http://127.0.0.1:9000/mcp'

# 封装客户端调用服务器函数 --- 异步转同步
def call_mcp_tool(tool_name: str, **kwargs):
    async def _call():
        async with Client(MCP_SERVER_URL) as client:
            return await client.call_tool(tool_name, kwargs)

    return asyncio.run(_call())


# 测试
if __name__ == '__main__':
    print(call_mcp_tool("verify_code", receiver="3503294776@qq.com", code="1111"))
```

### 调用链路追踪

```
LangChain 工具函数（LoadTools.py）
    ↓ 调用
call_mcp_tool("find_email", sql="SELECT...", params=["cc"])
    ↓ 创建异步上下文
fastmcp.Client("http://127.0.0.1:9000/mcp")
    ↓ HTTP POST (JSON-RPC)
MCP Server 路由到 find_email 函数
    ↓ 执行
MySQL 查询 → 返回结果
    ↓ 原路返回
MCP Server → Client → call_mcp_tool → LangChain 工具 → Agent
```

### 关键设计：异步转同步

```
MCP Client 使用的是 fastmcp.Client，它是异步的（async/await）
但 LangChain 的工具函数是同步的

解决方案：asyncio.run()
  • 内部创建异步函数 _call()
  • 通过 asyncio.run() 将异步调用转为同步调用
  • 对 LangChain 来说，工具函数就是普通的同步函数

注意：还使用了 warnings.filterwarnings 来忽略 DeprecationWarning
```

### 代码分析：`myagent/CreateAgent.py`

```python
import json

from langchain.agents import create_agent
from mymodel.LoadModel import load_model
from mytools.LoadTools import find_email, send_email, verify_code, find_data
from myutils.ResponseUtil import response_util


def init_agent():
    return create_agent(
        model=load_model(),
        system_prompt="""
        你是一个多功能业务智能体 (Business Agent)。

        你有两类能力：
        A. 账户邮箱验证码业务  
        B. 图数据库医疗知识检索业务 (RAG)

        你必须根据用户输入选择正确的流程，且不得混用流程。

        ==================================================
        一、流程 A：账户邮箱验证码发送与校验
        ==================================================

        1. 当用户输入包含"请给某个名字发送邮件"或类似表达时：
           - 将某个名字视为用户名(username)。
           - 进入【邮箱验证码发送流程】。
        2. 邮箱验证码发送流程：
           ① 调用 find_email 工具  
              - 参数：username
              - 返回：email  
           ② 若 email 不存在 → 回复：
              "未找到该用户名绑定的邮箱，请确认用户名是否正确。"
           ③ 若 email 存在 → 调用 send_email 工具  
              - 参数：receiver=email  
           ④ 发送成功 → 回复：
              {
                "code": 200,
                "msg: "发送成功",
                "data": email 邮箱号
              }
        3. 当用户输入包含"请检验验证码"或"验证码是否正确"时：
           - 提取邮箱号(email)与验证码(code)
           - 调用 verify_code 工具  
             - 参数：key=email, code=验证码  
           - result=success → 回复：
             "验证码正确，验证通过。"
           - result=fail → 回复：
             "验证码错误或已过期，请重新获取。"
        ==================================================
        二、流程 B：Neo4j 图数据库知识检索 (RAG)
        ==================================================
        4. 当用户提出查询医疗业务知识、关系数据、图谱信息等问题时：
           - 把用户问题转为 Cypher 查询语句，只把用户问题中的内容生成 Cypher 查询语句，不需要不相关的内容
           比如：请帮我查询适合糖尿病的菜肴，对应的 Cypher 查询语句
           MATCH p = (d:Disease {name: '糖尿病'})-[:DISEASE_DISHES]->(di:Dishes) RETURN p
           - 只允许生成 MATCH / RETURN 类型 Cypher
        5. 调用 find_data 工具  
           - 参数：
                cypher: 生成好的 Cypher
                params: 如需参数则提供
        6. 工具返回 result 后：
           - 用自然语言总结查询结果回复用户
           - 不展示 Cypher 原文
           - 若 result 为空 → 回复"未查询到相关数据"
        ==================================================
        三、异常处理
        ==================================================
        - find_data 返回 error → 回复："图数据库查询失败，请稍后重试。"
        - 其它工具异常按各自流程规则处理
        ==================================================
        四、行为约束
        ==================================================
        - 不混淆两类流程  
        - 不在验证码流程中调用 find_data  
        - 不在知识检索流程中调用 send_email / verify_code  
        - 不编造数据库或邮件结果  
        - 不输出多余解释
        ==================================================
        """,
        tools=[find_email, send_email, verify_code, find_data],
        debug=True,
    )


if __name__ == '__main__':
    agent = init_agent()
    response = agent.astream(
        {"messages": [{"role": "user", "content": "hello"}]}
    )
    print(response_util(response))
    result = json.loads(response_util(response))
    print(result)
    print(result['data'])
```

**核心要素：**
- `model` — 大语言模型（Qwen 3.5 Plus）
- `system_prompt` — 系统提示词（定义 Agent 的行为规则）
- `tools` — 可用工具列表（4 个工具函数）
- `debug=True` — 开启调试模式，方便查看 Agent 的思考过程

## 3.3 工具包装与文档编写

### 代码分析：`mytools/LoadTools.py`

工具包装层负责将 MCP 调用包装为 LangChain 兼容的工具函数，并通过**函数文档字符串（docstring）**告诉 Agent 如何使用。

```python
from mymcp.MCPClient import call_mcp_tool
from typing import Dict, Any, List


# ===================== 数据库工具 =====================
def find_email(sql: str, params: str | None = None) -> Dict[str, Any]:
    """
    查询数据库中的数据
    数据库包含一张表：user（用户表），字段如下：
        - username：用户名
        - email：邮箱号
        - 支持参数化查询，如：`SELECT * FROM user WHERE username=%s`
    """
    return call_mcp_tool(
        tool_name="find_email",
        sql=sql,
        params=params or []
    )

# ===================== 邮件工具 =====================
def send_email(receiver: str) -> Dict[str, Any]:
    """
    给对应的邮箱号发送验证码工具
    """
    return call_mcp_tool(
        tool_name="send_email",
        receiver=receiver
    )

# ===================== 验证码工具 =====================
def verify_code(receiver: str, code: str) -> Dict[str, Any]:
    """
    验证码验证工具
    """
    return call_mcp_tool(
        tool_name="verify_code",
        receiver=receiver,
        code=code
    )

# ===================== neo4j工具 =====================
def find_data(cypher: str, params: dict | None = None) -> Dict[str, Any]:
    """
    如果涉及到疾病信息查询，先查询neo4j数据库中的数据
    只查询用户问题相关的内容信息
    关于疾病查询的数据存储的关系信息如下：
        | 关系类型                 | 含义说明          | 指向节点         |
        | -------------------- | ------------- | ------------ |
        | `DISEASE_ACOMPANY`   | 疾病的并发症 / 伴随疾病 | `Disease`    |
        | `DISEASE_CATEGORY`   | 疾病所属类别        | `Category`   |
        | `DISEASE_CHECK`      | 疾病相关检查项目      | `Check`      |
        | `DISEASE_CUREWAY`    | 疾病治疗方式        | `Cureway`    |
        | `DISEASE_DEPARTMENT` | 就诊科室          | `Department` |
        | `DISEASE_DISHES`     | 适合疾病的菜肴       | `Dishes`     |
        | `DISEASE_DO_EAT`     | 推荐进食的食物       | `Food`       |
        | `DISEASE_NOT_EAT`    | 不推荐进食的食物      | `Food`       |
        | `DISEASE_DRUG`       | 治疗或相关药物       | `Drug`       |
        | `DISEASE_SYMPTOM`    | 疾病症状          | `Symptom`    |
    关于疾病查询的节点信息如下：
        | 节点标签         | 含义       |
        | ------------ | -------- |
        | `Disease`    | 疾病（核心节点） |
        | `Category`   | 疾病分类     |
        | `Check`      | 检查项目     |
        | `Cureway`    | 治疗方式     |
        | `Department` | 科室       |
        | `Dishes`     | 菜肴       |
        | `Food`       | 食物       |
        | `Drug`       | 药物       |
        | `Symptom`    | 症状       |
    """
    return call_mcp_tool(
        tool_name="find_data",
        cypher=cypher,
        params=params or {}
    )
```

## 3.4 Agent 决策流程解析

### 完整的决策过程

以"糖尿病有什么症状？"为例：

```
步骤 1：用户输入
    "糖尿病有什么症状？"

步骤 2：LLM 分析
    → 这是一个医疗知识查询（属于流程 B）
    → 需要查询 Neo4j 图数据库
    → 需要使用 find_data 工具

步骤 3：LLM 生成 Cypher
    → 根据 find_data 的文档 Schema
    → 疾病 → 症状 的关系是 DISEASE_SYMPTOM
    → 生成 Cypher：
      MATCH (d:Disease)-[:DISEASE_SYMPTOM]->(s:Symptom)
      WHERE d.name = '糖尿病'
      RETURN s.name

步骤 4：调用工具
    → find_data(cypher="MATCH...", params={})
    → MCP Client → MCP Server → Neo4j
    → 返回：{"result": [{"s.name": "多饮"}, {"s.name": "多尿"}, ...]}

步骤 5：LLM 整合
    → "根据查询结果，糖尿病的主要症状包括：
       多饮、多尿、体重下降、视力模糊..."

步骤 6：返回用户
    → 通过 FastAPI 的 StreamingResponse 流式输出
```

### 多步推理示例

```
用户请求："请给cc发送邮件"

步骤 1：Agent 分析
    → 这是一个邮箱验证码业务（属于流程 A）
    → 需要先查用户邮箱，再发送验证码

步骤 2：调用 find_email
    → find_email(sql="SELECT * FROM user WHERE username=%s", params=["cc"])
    → 返回：{"result": [{"username": "cc", "email": "cc@example.com"}]}

步骤 3：Agent 继续分析
    → 已获取邮箱为 cc@example.com，需要发送验证码

步骤 4：调用 send_email
    → send_email(receiver="cc@example.com")
    → 返回：{"message": "邮件发送成功！"}

步骤 5：Agent 整合回答
    → 返回 JSON 格式：
      {
        "code": 200,
        "msg": "发送成功",
        "data": "cc@example.com"
      }
```

│   DISEASE_NOT_EAT    → 不推荐进食的食物                        │
│   DISEASE_DRUG       → 治疗或相关药物                          │
│   DISEASE_SYMPTOM    → 疾病症状                               │
│                                                             │
│   节点类型汇总：                                              │
│   Disease | Category | Check | Cureway | Department         │
│   Dishes | Food | Drug | Symptom                            │
└─────────────────────────────────────────────────────────────┘

## 4.2 自然语言转 Cypher 查询

### 这是 Agentic RAG 的核心能力

Agent 的 LLM 根据工具文档中描述的 Schema，自动将自然语言转换为 Cypher 查询。

### 常见查询模式

| 用户问题 | LLM 生成的 Cypher 查询 |
|----------|----------------------|
| "糖尿病有什么症状？" | `MATCH (d:Disease)-[:DISEASE_SYMPTOM]->(s:Symptom) WHERE d.name = '糖尿病' RETURN s.name` |
| "高血压用什么药？" | `MATCH (d:Disease)-[:DISEASE_DRUG]->(dr:Drug) WHERE d.name = '高血压' RETURN dr.name` |
| "糖尿病看哪个科室？" | `MATCH (d:Disease)-[:DISEASE_DEPARTMENT]->(dep:Department) WHERE d.name = '糖尿病' RETURN dep.name` |
| "糖尿病不能吃什么？" | `MATCH (d:Disease)-[:DISEASE_NOT_EAT]->(f:Food) WHERE d.name = '糖尿病' RETURN f.name` |
| "糖尿病有什么并发症？" | `MATCH (d:Disease)-[:DISEASE_ACOMPANY]->(comp:Disease) WHERE d.name = '糖尿病' RETURN comp.name` |
| "糖尿病需要做什么检查？" | `MATCH (d:Disease)-[:DISEASE_CHECK]->(c:Check) WHERE d.name = '糖尿病' RETURN c.name` |
| "适合糖尿病的菜肴有哪些？" | `MATCH (d:Disease {name: '糖尿病'})-[:DISEASE_DISHES]->(di:Dishes) RETURN di.name` |

### Text-to-Cypher 的关键要素

```
┌─────────────────────────────────────────────────────────────┐
│          Agent 如何正确生成 Cypher？                          │
│                                                             │
│  1. 📖 Schema 理解                                          │
│     Agent 从 find_data 工具的 docstring 中学习：              │
│     - 10 种关系类型（DISEASE_SYMPTOM, DISEASE_DRUG, ...）     │
│     - 9 种节点类型（Disease, Symptom, Drug, ...）            │
│                                                             │
│  2. 🔍 实体识别                                              │
│     "糖尿病" → Disease 节点                                  │
│     "症状" → Symptom 节点 + DISEASE_SYMPTOM 关系             │
│                                                             │
│  3. 🗺️ 路径规划                                              │
│     起点(Disease) → 关系(DISEASE_SYMPTOM) → 终点(Symptom)    │
│                                                             │
│  4. 📝 查询生成                                              │
│     MATCH (起点)-[:关系]->(终点) WHERE 条件 RETURN 目标       │
│                                                             │
│  5. 🛡️ 安全约束（来自 System Prompt）                        │
│     只生成 MATCH/RETURN，不生成 CREATE/DELETE/SET            │
│     只查询用户问题相关的内容，不查询不相关的内容                │
└─────────────────────────────────────────────────────────────┘
```

## 4.3 邮件验证码服务实现

### 流程 A 的完整数据流

```
┌──────────────────────────────────────────────────────────────────┐
│                    邮箱验证码服务流程                               │
│                                                                  │
│  用户请求："请给cc发送邮件"                                         │
│      │                                                           │
│      ▼                                                           │
│  ┌──────────────┐                                                │
│  │ 步骤1：查询邮箱 │  find_email(sql="SELECT * FROM user            │
│  │              │            WHERE username=%s", params=["cc"])   │
│  │  MySQL →     │  → 返回: {"result": [{"username":"cc",          │
│  │  user 表      │            "email":"cc@example.com"}]}          │
│  └──────┬───────┘                                                │
│         │                                                        │
│         ▼                                                        │
│  ┌──────────────┐                                                │
│  │ 步骤2：发送邮件 │  send_email(receiver="cc@example.com")         │
│  │              │  → 生成 4 位随机码（每位 0-9 随机）               │
│  │  SMTP →      │  → QQ 邮箱 SMTP (smtp.qq.com:587) TLS 发送     │
│  │  QQ 邮箱      │  → Redis 存储: key=receiver邮箱, value=验证码   │
│  │  Redis →     │     ex=60 (TTL 60 秒自动过期)                   │
│  │  验证码缓存    │  → 返回: {"message": "邮件发送成功！"}          │
│  └──────┬───────┘                                                │
│         │                                                        │
│         ▼                                                        │
│  ┌──────────────┐                                                │
│  │ 步骤3：验证码校验│  verify_code(receiver="cc@example.com",        │
│  │              │              code="5832")                       │
│  │  Redis →     │  → 从 Redis 读取 r.get(receiver)               │
│  │  验证码比对    │  → 比对用户输入 vs 存储的验证码                   │
│  │              │  → 返回: {"result": "验证码验证成功！"}           │
│  └──────────────┘                                                │
│                                                                  │
│  安全机制：                                                       │
│  • 验证码 4 位数字，使用 random.randint(0, 9) 随机生成             │
│  • Redis set(key, value, ex=60) — 60 秒自动过期                   │
│  • SMTP 使用 starttls() 启用 TLS 加密                            │
│  • 发件人使用 QQ 邮箱授权码（非明文密码）                            │
└──────────────────────────────────────────────────────────────────┘
```

## 4.4 FastAPI 接口设计与 SSE 流式输出

### 代码分析：`main.py`

```python
import asyncio
import json
from fastapi import FastAPI, Request
from contextlib import asynccontextmanager
from starlette.responses import StreamingResponse
from myagent.CreateAgent import init_agent
from fastapi.middleware.cors import CORSMiddleware
from myutils.ResponseUtil import response_util


# ===================== 启动项目和关闭项目执行 =====================
@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动时执行
    print("服务器启动，初始化智能体等资源...")
    app.state.agent = init_agent()  # 全局智能体对象
    yield  # ---- 这里之后是运行期  ----
    # 关闭时执行
    print("服务器关闭，释放资源...")


app = FastAPI(lifespan=lifespan)

# ===================== 跨域配置 =====================
origins = ["http://127.0.0.1:8080", ]

app.add_middleware(
    CORSMiddleware,
    allow_origins=origins,
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)


# ===================== 发送验证码接口 =====================
@app.get("/sendEmail/{username}")
def send_email(request: Request, username: str):
    agent = request.app.state.agent
    try:
        response = agent.invoke(
            {"messages": [{"role": "user", "content": f"请给{username}发送邮件"}]}
        )
        result = json.loads(response_util(response))
        return {
            "code": 200,
            "msg": "发送成功",
            "data": result['data'],
        }
    except Exception as e:
        print(f"发送验证码错误：{e}")
        return {
            "code": 500,
            "msg": "服务器内部错误",
        }


# ===================== 验证验证码接口 =====================
@app.get("/verifyCode/{receiver}/{code}")
def verify_code(request: Request, receiver: str, code: str):
    agent = request.app.state.agent
    try:
        agent.invoke(
            {"messages": [{"role": "user", "content": f"请检验验证码，邮箱号为{receiver}，验证码为{code}"}]}
        )
        return {
            "code": 200,
            "msg": "验证成功",
        }
    except Exception as e:
        return {
            "code": 500,
            "message": "服务器内部错误",
        }


# ===================== 聊天接口 =====================
@app.get("/chat")
def chat(request: Request, messages: str):
    agent = request.app.state.agent
    try:
        response = agent.invoke({
            "messages": [{"role": "user", "content": messages}]
        })
        result = response_util(response)
        def streaming_content():
            for i in result:
                yield f"data: {json.dumps({'content': i})}\n\n"
            yield "data: [DONE]\n\n"
        return StreamingResponse(streaming_content(), media_type="text/event-stream")
    except Exception as e:
        return {
            "code": 500,
            "msg": "服务器内部错误",
        }
```

### 代码分析：`myutils/ResponseUtil.py`

```python
from langchain_core.messages import AIMessage


def response_util(response):
    """
    从 LangChain 的 result 中安全提取最后一条 AI 回复内容。
    如果没有 AIMessage，则返回空字符串。
    """
    messages = response.get("messages", [])
    if not messages or not isinstance(messages, list):
        return ""

    ai_contents = [
        msg.content
        for msg in messages
        if isinstance(msg, AIMessage) and msg.content
    ]

    # 安全返回最后一条
    if ai_contents:
        return ai_contents[-1]
    else:
        return "模型没有输出"
```

### API 接口总览

| 接口 | 方法 | 功能 | 返回格式 |
|------|------|------|----------|
| `/sendEmail/{username}` | GET | 发送验证码（调用 Agent 执行流程 A） | JSON |
| `/verifyCode/{receiver}/{code}` | GET | 验证验证码（调用 Agent 执行流程 A） | JSON |
| `/chat?messages=...` | GET | 通用对话（调用 Agent 执行流程 B） | SSE 流式 |

### 关键设计要点

```
🔧 生命周期管理（lifespan）：
   • 使用 @asynccontextmanager 装饰器
   • 启动时初始化 Agent：app.state.agent = init_agent()
   • Agent 是全局单例，所有请求共用同一个 Agent 实例
   • 关闭时执行清理逻辑

🔄 跨域配置（CORS）：
   • 允许来自 http://127.0.0.1:8080 的跨域请求
   • allow_credentials=True：允许携带认证信息
   • 支持所有 HTTP 方法和请求头

📡 SSE 流式输出（/chat 接口）：
   • 使用 StreamingResponse + text/event-stream
   • 遍历 response_util 返回的字符串，逐字符输出
   • 格式：data: {"content": "单个字符"}\n\n
   • 结束时发送：data: [DONE]\n\n

🔗 Agent 调用方式：
   • agent.invoke()：同步调用，等待完整结果
   • 传入格式：{"messages": [{"role": "user", "content": "..."}]}
   • 结果通过 response_util 提取最后一条 AI 消息
```
